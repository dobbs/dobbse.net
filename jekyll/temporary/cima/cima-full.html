    • ❑ Open Source & Government
      • ❑ buy vs. build is a continuum
        • ❑ When you buy, you end up having to pay someone to customize it -- especially in government
          • ❑ vendors sell nationally and internationally
          • ❑ in this audience you're concerned primarily with national and especially state laws
          • ❑ even when you buy you are also paying for custom software -- the theory is that you pay less because the basic application already handles the common cases
          • ❑ for complex systems you probably are also paying for training and documentation
        • ❑ When you build your cost is all developer salaries or consulting fees
        • ❑ It's actually hard to tell which of those extremes is more cost effective
          • ❑ Both models require similar up front investment of time to investigate potential solutions
          • ❑ The proprietary vendors will take you out to lunch. Probably no one in the open source project will be doing that. You have to judge for yourself if the lunch they buy you is worth the hour of your undivided attention they get in exchange.
        • ❑ Open source lands squarely in-between those extremes
          • ❑ you start with an existing code base which has been fairly well tested in some real-world applications
          • ❑ in many cases it isn't very well documented and training may be hard to come by -- the cost is in the time it takes your developers to learn to use the tool on their own, or paying some consultant to do the training
          • ❑ some dimensions of your problem will be well addressed or even solved by these tools -- same story as if you bought a commercial application
          • ❑ but it hasn't been deployed in your particular application
          • ❑ you will find things in the code that don't work for your application without some customization
          • ❑ you can code around the limitations in your own code as you would with any 3rd party solution
          • ❑ you can also change their code so it works right for your problem
          • ❑ give your changes back
            • ❑ if you give your changes back to the project and the project accepts them, others will benefit from your pain
            • ❑ you won't have to take on the additional pain of maintaining your patches separately from the open source source tree.
      • ❑ taking license - GPL, LGPL, BSD, MIT, Apache, Artistic
        • ❑ Example of Apple Computer
          • ❑ Darwin is the unix underbelly of MacOS -- it uses a Mach Microkernel and all the unix command line utilities are derived from the BSD unix family.
          • ❑ One major reason Apple chose BSD rather than Linux is about the licensing. Linux is licensed under the GPL which has a viral clause in it -- any derived code must also be released under the GPL. By contrast the BSD family lives under a BSD license which allows you to redistribute the code under other licensing terms if you so choose -- that is more business friendly.
          • ❑ Apple's legendary ease of use still lives in proprietary source code and single-vendor hardware.
          • ❑ Apple enjoys the benefit of years of production-tested code from the BSD family while still keeping enough of their code proprietary to provide compelling reasons for customers to buy Apple hardware. Also, as the BSD crowd find and fix bugs in their utilities, Apple can incorporate those fixes without having to pay for all those developers' salaries and benefits.
        • ❑ JBoss vs. Geronimo example
          • ❑ JBoss is LGPL -- you can build apps to run in JBoss without restriction, but if you want to release anything derived from JBoss it must also be licensed under LGPL
          • ❑ Geronimo is Apache 2.0 -- modify and release it under other licenses, but give credit to the copyright holders
      • ❑ about the cost
        • ❑ there are no license fees
        • ❑ but, you do have to invest time to learn something new
        • ❑ keeping up with the latest version gives the benefit of code maintained by someone else, but there's still work in keeping up
        • ❑ if you choose not to keep up with the latest version, there may still be work in maintaining your own patches to the code.
    • ❑ Web Applications
      • ❑ database applications with a web browser as the client
      • ❑ advantages of web applications
        • ❑ deployment - install once, run anywhere
        • ❑ maintenance - everyone is on the same version, you can change everyone's version by updating one server (or maybe a small cluster)
        • ❑ limited bag-of-tricks for the UI
          • ❑ much lower training cost for users
            • ❑ your employees have already used web-based forms at Amazon, or eBay
            • ❑ they also mostly know how to use hyperlinks and maybe even back buttons
          • ❑ lower development cost
            • ❑ smaller pallet means UI decisions can move somewhat more quickly
            • ❑ sorta
              • ❑ there are still UI trade offs -- links vs forms, url designing, CSS, browser inconsitancies
      • ❑ disadvantages
        • ❑ limited bag-of-tricks for the UI
          • ❑ if you try to make HTML do tree widgets and combo-boxes, you can sink a lot of time in UI twidling, but you'll probably learn a lot about JavaScript and the DOM in the process.
          • ❑ statelessness of HTTP makes it harder to split long forms into multiple pages -- not impossible, but harder
        • ❑ network lag, redraws & blinks
          • ❑ can I demonstrate this?
        • ❑ design challenge: accommodating web conventions like hyperlinks, bookmarks, and the back button
        • ❑ design challenge: is it a web site or is it an application?
      • ❑ advanced practices
        • ❑ web logs for troubleshooting
          • ❑ Incredibly valuable in tracking down software problems -- there's a built-in audit trail of what users are doing on the system. We have our systems notify us of errors. In some cases we have fixed the problem before a user reports the error. In the case of user error, we know what the user was doing on the system and can coach them how to avoid the problem in the future.
        • ❑ controlled beta testing
          • ❑ Yahoo is doing some live beta testing of a new system. They divert some users to the beta version based on their cookie. They get immediate feedback on the beta and can throttle the load on the beta by controlling the cookies that get diverted to the new systems.
        • ❑ automated UI and workflow testing can be very easy
          • ❑ show communications.btest
        • ❑ live updates, continual development
          • ❑ We have a serious production application where our customers are managing their investments. People are keeping track of their money through us -- we have to be disciplined about how we manage this system. We update the software live almost every week. Live updates. The application is served by a pair of web servers. We take one out of the load balancing and upgrade it, then switch the load balancing to upgrade the other one. Not only does it allow us to continually improve our services, we also get a regular built-in test of our load balancing and failover.
        • ❑ other benefits of stateless protocol
          • ❑ key trick is to avoid session variables
          • ❑ pretty easy to throw hardware at scalability and failover problems.
          • ❑ If you use session variables, you're fighting against HTTP -- if your site volume grows dramatically, you'll have more complex problems in distributed memory or load balancing or failover.

    • ❑ Web Services
      • ❑ All web applications are a kind of web service
        • ❑ You may not be in REST or SOAP heaven for a while, but every web app you build can be potentially reused as part of a bigger Service Oriented Architecture. Clean XHTML with CSS for the pretty layout will go a very long way in making the transition to an SOA easier.
      • ❑ URLs are your method calls, everything in the form or after the question mark are arguments to the remote methods
      • ❑ page contents are structured return values, and return status can also indicate success or failure
      • ❑ XHTML is XML
      • ❑ CSS can provide the UI for browser based clients
